# 配合 v1.1 `extra_content` 的前端 UI 改造指南

> **受众**:WebChat Dashboard(前端)开发者,准备在
> `astrbot_plugin_ask_user_choice` v1.1.0 发版后改进选项框 UI。
>
> **目的**:让前端**利用** `extra_content` 字段(LLM 给用户看的 Markdown 补充说明),
> 把选项框从"问题+按钮"升级为"问题+推荐/理由+按钮",而不是把 `extra_content` 默默丢在
> `spec` 里不渲染。
>
> **必读前置 spec**:
> - `docs/superpowers/specs/2026-07-07-extra-content-field-amendment.md` — v1.1 数据契约
> - `docs/superpowers/specs/2026-07-02-blocking-interactive-choice-design.md` §5 — v1.0 前端架构

---

## 0. TL;DR

后端已经在 v1.1.0 把 `extra_content` 字段(可选 string,≤5000 字符,Markdown)
加到了 SSE 推出来的 `spec` dict 里。前端只需要做**两件事**:

1. **类型 + 截断**:`InteractiveChoicePart` 加 `extra_content?: string`,
   `truncateInteractiveChoice` 同步加 `EXTRA_CONTENT_MAX = 5000` 防御。
2. **渲染**:在 `InteractiveChoiceBox` 组件里,`prompt` 与 `options` 之间插入一个
   `prose` 区,用你仓库现成的 Markdown 渲染器(推荐 `markdown-it`,
   配 `linkify` + 严格 `sanitize`)把 `extra_content` 渲染出来。

后端**已发版并就绪**;前端**可以独立发版**,无顺序依赖(后端先合,前端后合)。
`[key: string]: unknown` 已存在,所以**纯类型层**不更新也能跑——只是不渲染新字段
(graceful degradation)。

---

## 1. 这次改动给前端带来的实际影响

### 1.1 数据层

`spec` dict(从 SSE `data` JSON 字符串解析得到)多了一个**可选**字段:

| 字段 | 类型 | 必填 | 长度上限 | 内容 | 是否持久化 |
|---|---|:---:|:---:|---|:---:|
| `extra_content` | `string` | ❌ | 5000 字符 | LLM 写的 Markdown 补充说明 | ✅ 进 bot 消息 parts 数组 |

> 持久化是 v1.1 wire format 升级的副作用:`_push_to_webchat_back_queue` 把事件改走
> `chain_type="interactive_choice"` 通道(`type="plain"` + `chain_type="interactive_choice"`
> + JSON 字符串 `data`),`chat_service.BotMessageAccumulator` 会把它落到 bot 消息的
> `parts` 里——所以刷新页面后 `extra_content` 会随消息一起恢复,**不需要**额外
> localStorage 逻辑。

### 1.2 渲染层

`InteractiveChoiceBox` 当前(假设)只渲染 `prompt` + `options` + 自由输入框。
v1.1 之后,**建议**在 `prompt` 与 `options` 之间插入一个 prose 区域展示
`extra_content`(没有就完全跳过,不留空块)。

### 1.3 后端**不会**做但**希望**前端做的

- Markdown 解析(后端原样透传,不做任何转义)
- XSS 防护 / URL sanitize
- 折叠 / TOC / 代码块高亮等"漂亮"工作
- 移动端排版适配

## 2. 数据契约(必读,所有代码改动以此为准)

### 2.1 完整 wire-format 样例

**SSE 推到 `back_queue` 的事件**(从 `tests/test_ask_user_choice_tool.py` 验证过的实际格式):

```json
{
  "type": "plain",
  "chain_type": "interactive_choice",
  "message_id": "<chat_service 的 SSE 流 uuid>",
  "data": "{\"request_id\":\"<plugin uuid>\",\"spec\":{\"type\":\"interactive_choice\",\"prompt\":\"选一个部署方案?\",\"options\":[{\"id\":\"A\",\"label\":\"蓝绿\",\"description\":\"零停机,成本翻倍\"},{\"id\":\"B\",\"label\":\"灰度\",\"description\":\"可控,需 LB 支持\"},{\"id\":\"C\",\"label\":\"滚动\",\"description\":\"最便宜,有中断窗口\"}],\"title\":\"部署方案\",\"input_placeholder\":\"可选:输入自定义方案\",\"extra_content\":\"**推荐 B**。\\n\\n理由:\\n- 兼顾成本与风险\\n- LB 已就绪\\n\\n**注意**: 灰度比例建议从 5% 起步\"},\"expires_at\":1700000000.0,\"umo\":\"webchat:FriendMessage:webchat!alice!sess\"}"
}
```

**前端解析后**得到 `InteractiveChoicePart`(就是你 store 里 `addChoice(part)` 的参数):

```typescript
const part: InteractiveChoicePart = {
  type: "interactive_choice",
  request_id: "<uuid>",          // 后端给,前端只用于 REST 提交 / store key
  prompt: "选一个部署方案?",
  title: "部署方案",
  options: [
    { id: "A", label: "蓝绿", description: "零停机,成本翻倍" },
    { id: "B", label: "灰度", description: "可控,需 LB 支持" },
    { id: "C", label: "滚动", description: "最便宜,有中断窗口" },
  ],
  input_placeholder: "可选:输入自定义方案",
  expires_at: 1700000000,

  // ── v1.1 新增(可选) ──
  extra_content:
    "**推荐 B**。\n\n理由:\n- 兼顾成本与风险\n- LB 已就绪\n\n**注意**: 灰度比例建议从 5% 起步",
};
```

### 2.2 TypeScript 类型扩展

**文件**:`dashboard/src/composables/parseInteractiveChoice.ts`

```typescript
export interface InteractiveChoicePart {
  type: "interactive_choice";
  request_id: string;
  prompt: string;
  title?: string;
  options: InteractiveChoiceOption[];
  input_placeholder?: string;
  expires_at?: number;

  // ── v1.1 新增 ──
  /**
   * 选填,Markdown 文本。LLM 给用户看的推荐/理由/注意事项/优缺点对比。
   * 建议在 prompt 与 options 之间渲染为 prose(列表/代码块/链接/行内强调都允许);
   * 禁止渲染图片(防 LLM 注入不安全 URL)。
   * 长度上限 5000 字符,后端已截断,前端 truncateInteractiveChoice 也会二次截断。
   * 注意:此字段**不会**回传到 LLM tool result。
   */
  extra_content?: string;

  [key: string]: unknown;
}
```

> `[key: string]: unknown` 已经存在(v1.0 spec §5.1),所以**严格来说**不改类型也能跑——
> 旧前端拿到的 `extra_content` 字段会被吞掉(`unknown` 不会暴露给组件)。
> 但**类型层**必须更新,否则组件读 `part.extra_content` 会 TS 报错。

### 2.3 截断函数同步

**文件**:`dashboard/src/composables/parseInteractiveChoice.ts`

```typescript
const LIMITS = {
  PROMPT_MAX: 200,
  TITLE_MAX: 30,
  LABEL_MAX: 30,
  DESC_MAX: 200,
  PLACEHOLDER_MAX: 60,
  EXTRA_CONTENT_MAX: 5000,  // v1.1 新增
};

export function truncateInteractiveChoice(part: InteractiveChoicePart): InteractiveChoicePart {
  let mutated = false;
  const out: InteractiveChoicePart = { ...part };
  // ... (原有 prompt/title/label/desc/placeholder 截断逻辑保持不变) ...

  // v1.1 新增
  if (typeof out.extra_content === "string" && out.extra_content.length > LIMITS.EXTRA_CONTENT_MAX) {
    out.extra_content = out.extra_content.slice(0, LIMITS.EXTRA_CONTENT_MAX);
    mutated = true;
  }
  return mutated ? out : part;
}
```

后端已经做过**第一重**截断,这里只是防御性兜底,正常情况不会触发。

### 2.4 校验函数**不需要**改

`validateInteractiveChoice` 只检查 `request_id` / `prompt` / `options` 三项,不会因为
多了 `extra_content` 而拒绝。LLM 写得再烂,只要 prompt+options 合法,前端的
`addChoice(part)` 仍会把它收下——Markdown 渲染出问题由 prose 组件自己负责。

## 3. UI 改造建议(不强约束,你仓库自己定)

### 3.1 布局建议

在 `InteractiveChoiceBox.vue` 里,`prompt` 区下方、`options` 按钮列表上方,插入一个 prose 区:

```vue
<template>
  <div class="interactive-choice-box">
    <!-- 1. prompt: 仍放最顶 -->
    <header v-if="part.title" class="title">{{ part.title }}</header>
    <p class="prompt">{{ part.prompt }}</p>

    <!-- 2. v1.1 新增:extra_content markdown 渲染区 -->
    <div
      v-if="part.extra_content"
      class="prose extra-content"
      v-html="renderedExtraContent"
    />

    <!-- 3. options 按钮列表 -->
    <div class="options">
      <button
        v-for="opt in part.options"
        :key="opt.id"
        :disabled="state !== 'pending'"
        @click="onOptionClick(opt)"
      >
        <span class="label">{{ opt.label }}</span>
        <span v-if="opt.description" class="description">{{ opt.description }}</span>
      </button>
    </div>

    <!-- 4. 自由输入 -->
    <input
      v-model="freeText"
      :placeholder="part.input_placeholder || '...'"
      :disabled="state !== 'pending'"
      @keyup.enter="onInputSubmit"
    />
  </div>
</template>
```

**布局选择**(选一个就行,不必多选):

- **A. 内联**(推荐 v1):把 prose 紧贴 prompt 下方,作为"问题的详细说明"。用户读完
  推荐理由再做决定——这正是 `extra_content` 字段名的字面意义。
- **B. 折叠区**:用一个 `<details>` 默认展开,适合内容长(> 1000 字符)时避免撑屏。
- **C. 侧栏**:桌面端右栏、移动端底部抽屉。工程量大,留到 v1.2+。

### 3.2 Markdown 库选择(推荐 `markdown-it`)

`Astrbot/dashboard` 现有依赖里一般已经有 `markdown-it`(如果装过)或
`marked`(更轻)。**任选其一**即可,优先沿用项目里已有的。

```ts
// composables/renderExtraContent.ts
import MarkdownIt from "markdown-it";

const md = new MarkdownIt({
  html: false,        // ← 关键:禁用原始 HTML,防 LLM 注入 <script>
  linkify: true,      // URL 自动转链接
  breaks: true,       // 单换行 → <br>
  typographer: false, // 不智能标点(LLM 中文混英文时容易翻车)
});

// 可选:自定义链接打开方式
const defaultLinkOpen = md.renderer.rules.link_open;
md.renderer.rules.link_open = (tokens, idx, options, env, self) => {
  const token = tokens[idx];
  const hrefAttr = token.attrIndex("href");
  if (hrefAttr >= 0) {
    const href = token.attrs![hrefAttr][1];
    if (/^javascript:/i.test(href)) {
      token.attrs![hrefAttr][1] = "#";
    }
    token.attrSet("target", "_blank");
    token.attrSet("rel", "noopener noreferrer");
  }
  return defaultLinkOpen!(tokens, idx, options, env, self);
};

export function renderExtraContent(mdText: string): string {
  return md.render(mdText);
}
```

**关键配置**:
- `html: false` — 拒绝原始 HTML,LLM 即便在 `extra_content` 里塞 `<script>` 也会被
  转义成纯文本(因为 `markdown-it` 把原始 HTML 视为内联 HTML 透传;`html: false`
  后会**转义**)。**这是 XSS 防御的第一道防线,不要去掉**。
- `linkify: true` — 裸 URL 自动成链接,体验更好。
- 链接强制 `target=_blank` + `rel="noopener noreferrer"` — 用户点链接不离开 dashboard。
- 拒绝 `javascript:` 协议 — 防 prompt injection 利用 markdown link 注入。

### 3.3 不渲染图片

**硬性约束**:`extra_content` 里的 `![](url)` **必须**显示为"[图片]"占位符或不显示,
**不要**渲染。原因:

- LLM 输出的 URL 不可信,可能是:
  - 跟踪像素(用户隐私)
  - 恶意图片(onerror 弹窗、SSRF)
  - LLM 幻觉的 URL(404 噪声)
- `markdown-it` 默认会渲染 `<img>`,需要**显式禁用**。

```ts
// 在 markdown-it 实例化后加:
md.renderer.rules.image = () => "[图片]";
// 或者更激进:根本不注册 image 渲染规则
```

### 3.4 可访问性(a11y)

- prose 容器加 `role="note"` 或 `aria-label="补充说明(由 AI 生成)"` —— 屏幕阅读器
  能区分"问题"和"AI 补充"。
- 长内容(> 1500 字符)考虑加 `aria-live="polite"` 让 SR 在更新时朗读。
- 代码块 `pre` 加 `tabindex="0"` 让键盘用户能滚动。

### 3.5 暗色模式

`markdown-it` 输出的 HTML 是语义标签(`<h1>` `<ul>` `<code>`),颜色由 CSS 决定。
如果 `prose` 类已有暗色变量,直接套上;没有的话,看下面的最小样式模板:

```css
.extra-content :is(h1, h2, h3) {
  font-weight: 600;
  margin: 0.5em 0;
}
.extra-content p {
  margin: 0.5em 0;
  line-height: 1.6;
}
.extra-content ul, .extra-content ol {
  padding-left: 1.5em;
}
.extra-content code {
  background: rgba(127, 127, 127, 0.15);
  padding: 0.1em 0.3em;
  border-radius: 3px;
  font-size: 0.9em;
}
.extra-content pre {
  background: rgba(127, 127, 127, 0.1);
  padding: 0.8em;
  border-radius: 6px;
  overflow-x: auto;
}
.extra-content a {
  color: var(--brand-color, #3b82f6);
  text-decoration: underline;
}
```

## 4. Pinia store / SSE 消费层**不需要**改

`stores/interactiveChoice.ts` 里的 `addChoice(part)` 只做"塞进 dict + 持久化",
`part` 整个被原样保存(`InteractiveChoicePart` 类型已经包含 `extra_content?: string`)。
**没有**字段级解构需要更新。

同理 `ChatMessageList.vue` 里的 SSE listener(在 v1.0 spec §5.4 已定义):

```ts
case 'interactive_choice': {
  const part: InteractiveChoicePart = {
    type: 'interactive_choice',
    request_id: event.data.request_id,
    ...event.data.spec,            // ← extra_content 会自动 spread 进来
    expires_at: event.data.expires_at,
  };
  store.addChoice(part);
  break;
}
```

`...event.data.spec` 会把 `extra_content` 自动带上,**无需**白名单字段。

## 5. 端到端验证清单(发版前必跑)

按以下顺序手测一遍,全部通过才能合并:

| # | 场景 | 期望 |
|---|---|---|
| 1 | LLM 调用 `ask_user_choice` 不带 `extra_content` | UI 行为**完全等同 v1.0**,无 prose 区出现 |
| 2 | LLM 调用时 `extra_content=""` 或纯空白 | UI 行为**完全等同 v1.0**(spec 字段被后端 strip 后丢弃) |
| 3 | LLM 传入正常 Markdown(标题+列表+代码+链接) | prose 区正确渲染,链接可点开新标签 |
| 4 | LLM 传入 `**推荐 A**` 之类纯粗体 | 粗体生效 |
| 5 | LLM 在 `extra_content` 塞 `<script>alert(1)</script>` | **不**弹窗,转义为可见文本 |
| 6 | LLM 塞 `[evil](javascript:alert(1))` | **不**触发,链接被替换为 `#` 或显示为纯文本 |
| 7 | LLM 塞 `![](http://evil.com/track.gif)` | 显示为"[图片]"或不显示,**不**发请求 |
| 8 | LLM 塞长度 > 5000 字符 | 显示截断版本(后端 + 前端双重截断,行为一致) |
| 9 | 页面 hard refresh 时 `extra_content` 仍显示 | ✅(因 v1.1 wire format 升级,bot message parts 已持久化) |
| 10 | 多 tab 同步:A 选完,B 刷新 | B 收不到 `extra_content`(已 resolved,prose 区消失) |
| 11 | 暗色 / 亮色模式切换 | prose 区颜色跟随,无白底黑字在暗色下刺眼 |
| 12 | 移动端宽度(prose 长文本自动换行) | 不撑出横向滚动条(除非是 `<pre>` 代码块) |

## 6. 边界与降级

### 6.1 旧前端(没更新) vs 新后端(v1.1.0)

| 后端版本 | 前端版本 | 行为 |
|---|---|---|
| v1.0 | v1.0 | 正常 |
| v1.0 | v1.1 | 正常(前端加的字段后端不写,等价 v1.0) |
| **v1.1** | v1.0 | **后端多写 `extra_content`,前端不渲染——graceful degradation**,与 v1.0 无差别 |
| **v1.1** | v1.1 | 完整体验 |

结论:**两个版本可独立发版**。先合后端,后合前端,顺序不限。

### 6.2 LLM 行为约束(给前端做展示用)

`extra_content` 是**纯展示**字段,**不会**回传到 LLM 的 tool result
(见 spec §5 `User selected: A (id=A)\nAdditional note: <free_text>`)。
这意味着:

- LLM 在同一次 `ask_user_choice` 调用里写过的内容**不会**出现在下一轮上下文。
- LLM 可能会在下一轮回复里**复述** `extra_content`(因为本来就在它自己上下文里)。
  这是设计预期,不是 bug。
- 如果 LLM 觉得"用户应该看到我的分析",鼓励它在 `extra_content` 里写,而不是:
  - 塞进 `prompt`(会被 200 字符截断)
  - 塞进某个 `options[].description`(会跟该选项语义错位)
  - 塞进自己的自然语言回复(浪费 token)

前端不需要做任何"防 LLM 复读"的处理。

### 6.3 旧 dashboard(没支持 v1.1 wire format) 的兼容性

如果前端 dashboard 还没升级到能解析 `chain_type="interactive_choice"` 通道
(v1.0 wire format 是 `type="interactive_choice"` + dict `data`),
那么 v1.1 后端推送的 `type="plain"` 事件**会被前端当成普通纯文本**——
显示成一段 JSON 字符串。这**会回退**到"问题+按钮"完全消失的体验。
**所以建议**:前端 PR 跟后端发版**同步**做;后端先合会让用户看到一段 JSON。

如果担心这点,后端可以提供配置开关 `wire_format_version: "v1" | "v1.1"`,
但本仓库 v1.1.0 **默认 v1.1**,不向后兼容 v1.0 wire format(见
`tests/test_ask_user_choice_tool.py::test_push_to_webchat_back_queue_uses_chain_type_envelope`)。

## 7. 推荐 PR 拆分(参考,不强求)

按依赖和可独立验证性,前端 PR 拆成 2~3 个小 PR 比较好 review:

### PR 1:类型 + 截断(无 UI 变化,~30 行)

- `composables/parseInteractiveChoice.ts`:`InteractiveChoicePart` 加字段 +
  `truncateInteractiveChoice` 加截断
- `composables/parseInteractiveChoice.test.ts`:加 2~3 个单测
  - `validateInteractiveChoice` 不拒绝含 `extra_content` 的 part
  - `truncateInteractiveChoice` 截断超长 `extra_content`
  - 缺省 `extra_content` 时 `part.extra_content === undefined`

**可独立合并**,仅类型层 + 防御性截断,用户**看不到**任何变化(组件没引用新字段)。

- 新建 `composables/renderExtraContent.ts`(markdown-it 封装,见 §3.2)
- 改 `InteractiveChoiceBox.vue`:插 prose 区 + `v-html="renderedExtraContent"`(§3.1)
- 加 CSS(§3.5 模板)
- 跑 §5 验证清单 1~12

依赖 PR 1。

### PR 3(可选):暗色模式 + a11y 优化

依赖 PR 2。

## 8. 已知/未实现(给前端看的 backlog)

- **代码块语法高亮**:markdown-it 本身不自带,需要 `markdown-it-highlightjs` 或
  `markdown-it-prism`。如果 LLM 经常贴 JSON/YAML/Python,可以加。
- **TOC(目录)**:长 `extra_content`(> 1500 字符且含多级标题)可以做。
  `markdown-it-toc-done-right` 插件,5 行代码搞定。
- **复制代码按钮**:在 `<pre>` 右上角加一个"复制"按钮,需要二次开发(纯 markdown-it
  不支持)。
- **LLM 主动高亮推荐项**:v1.1 spec 把"推荐项高亮"明确排到 v1.2+。LLM 现在只能在
  `extra_content` 文字里写"**推荐 B**",前端如果想做按钮变金/置顶,需要 LLM
  单独给个 `recommended_id` 字段——本仓库暂不提供。
